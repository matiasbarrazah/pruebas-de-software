# Ejercicio: Verificación vs. Validación con apoyo de IA

## Contexto general

En este ejercicio deberás analizar, diseñar, implementar y probar una aplicación sencilla a partir de un requerimiento incompleto y ambiguo, tal como ocurre frecuentemente en proyectos reales.

La actividad busca que distingan claramente entre verificación y validación, y que además demuestres capacidad de pensamiento crítico..

La tarea considera:

- Detectar ambigüedades y riesgos
- Tomar decisiones justificadas
- Diseñar pruebas útiles
- Identificar límites del sistema
- Evaluar críticamente el apoyo recibido por IA

## Enunciado del problema

Una pequeña empresa necesita una aplicación web para gestionar reservas de atención técnica de electrodomésticos.

El sistema debe permitir, al menos:

- Registrar clientes
- Registrar técnicos
- Agendar una reserva de atención técnica
- Cancelar una reserva
- Consultar las reservas futuras

El sistema lo utilizará la empresa de servicio técnico, que toma los pedidos por email y Whatsapp

La empresa menciona que desea una solución “simple, usable y confiable”, pero no entrega mayores detalles.

## Importante

El requerimiento anterior está deliberadamente incompleto.

Por lo tanto, parte del trabajo consiste en:

- Identificar ambigüedades
- Definir supuestos razonables
- Justificar decisiones
- Reflejar esas decisiones tanto en la implementación como en las pruebas

No todas las respuestas correctas serán iguales. Se evaluará fuertemente la coherencia entre lo que definan, lo que construyan y lo que prueben.

## Trabajo a realizar

### 1. Análisis inicial del requerimiento

A partir del enunciado entregado, el equipo debe:

- Identificar ambigüedades, vacíos o posibles conflictos del requerimiento.
- Proponer una versión mejorada y más precisa del requerimiento.
- Definir reglas de negocio mínimas para que el sistema pueda operar.
- Explicitar qué elementos quedarán fuera de alcance.

Como referencia, algunas preguntas que podrían aparecer son:

- ¿Una reserva puede quedar en el pasado?
- ¿Se permiten reservas superpuestas para un mismo técnico?
- ¿Qué datos mínimos requiere un cliente?
- ¿Qué restricciones deben cumplir usuarios y contraseñas?

No es obligatorio responder exactamente estas preguntas; pueden surgir otras adicionales.

### 2. Verificación vs. validación

El equipo debe explicar, aplicado a este caso, la diferencia entre:

- Verificación
- Validación

Además, deben indicar:

- Al menos 3 actividades de verificación concretas para este proyecto;
- Al menos 3 actividades de validación concretas para este proyecto.

No basta con definiciones teóricas genéricas. Deben estar de acuerdo al sistema desarrollado.

### 3. Registro de decisiones frente a ambigüedades

Deben incorporar una sección llamada:

**Decisiones de diseño frente a ambigüedades**

Para cada decisión relevante, registrar:

- Ambigüedad detectada
- Alternativas consideradas
- Decisión tomada
- Justificación
- Impacto en implementación
- impacto en pruebas

Se espera un mínimo de 5 decisiones relevantes.

### 4. Implementación del sistema

El equipo deberá construir una aplicación que implemente su propuesta.

#### Requisitos mínimos de implementación

- Aplicación web
- Persistencia de datos
- Manejo de errores
- Repositorio GitHub del proyecto.

#### Tecnologías

Puedes utilizar las tecnologías que estimen convenientes, siempre que el proyecto pueda ser ejecutado y revisado razonablemente.

Deben dejar instrucciones claras de instalación y ejecución en el README.md.

### 5. Estrategia, diseño y ejecución de pruebas

Deben definir una estrategia de pruebas coherente con el sistema implementado: ¿Quién y cómo se prueba?

La entrega debe incluir como mínimo:

- Objetivos de prueba;
- Alcance de pruebas;
- Criterios de entrada y salida;
- Evidencias de ejecución.

#### Casos obligatorios

Deben incluir, como mínimo:

- 8 casos de prueba funcionales;
- 5 casos de borde;
- 3 casos negativos o inválidos;
- 2 casos diseñados para tensionar o romper supuestos del sistema.

NOTA: Usar la planilla adjunta para registrar las pruebas (Manuales)/ SimpleTestCaseSuite_S1_2026_rev1.0.xlsx

### 6. Uso de IA

El uso de IA está permitido, pero debe ser declarado y analizado críticamente.

Deben incluir una sección llamada:

**Uso de IA en el proyecto**

Esta sección debe incluir, al menos:

- Herramientas utilizadas
- Para qué fueron usadas
- Qué decisiones no delegaron a IA y por qué.

No se evaluará negativamente el uso de IA en sí. Se evaluará negativamente el uso acrítico de IA.

### 7. Reflexión final

Deben incluir una breve reflexión técnica donde indiquen:

- el error o problema más importante detectado durante el desarrollo;
- cómo fue detectado;
- por qué no se detectó antes;
- qué mejorarían en su proceso de trabajo;
- una limitación actual del sistema;
- una mejora futura prioritaria.

### 8. Aspectos de trabajo en equipo y repositorio

Grupos de 2 personas.

El proyecto debe evidenciar trabajo ordenado.

Se pide:

- Repositorio GitHub del equipo
- Historial de commits consistente
- Uso de ramas. Gitflow u otro paradigma y por qué se escogió.
- Integración de tareas o seguimiento del trabajo;
- Evidencia de coordinación del equipo.

Si utilizan GitHub Projects, Issues, Pull Requests, Slack, Discord, Trello u otra herramienta, deben mostrarlo en la entrega.

No se espera un proceso perfecto, pero sí uno visible y justificable.

### 9. Entregables

La entrega debe incluir como mínimo:

- Repositorio GitHub con el código fuente.
- README.md con instrucciones de instalación y ejecución.
- Documento principal de análisis y pruebas en formato Markdown o PDF.
- Planilla de pruebas.
- Evidencias de pruebas (Logs por ejemplo).

#### 9.1 Estructura sugerida del documento principal

Se sugiere la siguiente estructura:

- Introducción
- Análisis del requerimiento
- Requerimiento mejorado
- Alcance y exclusiones
- Verificación vs. validación en el proyecto
- Decisiones de diseño frente a ambigüedades
- Diseño e implementación general
- Estrategia de pruebas
- Casos de prueba
- Pruebas que validan nuestros supuestos
- Uso de IA en el proyecto
- Reflexión final
- Conclusiones

### 10 Restricciones

- El trabajo debe ser desarrollado por el equipo (2 personas).
- La IA puede apoyar, pero no reemplaza la responsabilidad técnica del equipo.
- Todo supuesto importante debe quedar documentado.
- Toda prueba relevante debe poder ser entendida y trazada.
- Toda decisión importante debe ser defendible.

### 11. Criterios generales de evaluación

Se evaluará:

- Comprensión del problema
- Calidad del análisis
- Capacidad para detectar ambigüedades.
- Coherencia entre requerimientos, implementación y pruebas.
- Calidad de la estrategia de pruebas
- Capacidad de análisis crítico del uso de IA
- Orden, trazabilidad y calidad de la entrega.
